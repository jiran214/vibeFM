# 技术决策

## Web 框架

使用 Next.js。

## 网易云歌单导入

- CLI 只负责解析 `vibefm import <workspace> <url>` 和输出 JSON，导入业务位于 `core`。
- 使用网易云 `/api/v6/playlist/detail` 获取歌单元数据和完整 `trackIds`。
- 详情响应可能只携带部分 `tracks`，缺失歌曲通过 `/api/song/detail` 每批 100 首补齐，并按 `trackIds` 恢复歌单顺序。
- 同时兼容旧字段 `artists/album/duration/alias` 和新字段 `ar/al/dt/alia`。
- `playlist.json` 保存规范化数据，不直接持久化第三方接口的完整原始响应，降低外部字段变化对后续流程的影响。
- 使用同目录临时文件加重命名的方式原子更新 `playlist.json`。

## AI 请求

- AI 请求业务位于 `core`，不依赖 CLI、HTTP 或 UI。
- 使用官方 `openai` Node SDK，并统一通过 Chat Completions 接口请求。相比 Responses API，该接口更适合需要切换第三方 OpenAI 兼容 Base URL 的场景。
- 配置从项目根目录 `.env` 读取，必填项为 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL`。
- 启动进程中已存在的同名环境变量优先，`.env` 只提供缺失配置，便于 CI 和临时覆盖。
- `loadAiConfig` 负责加载及校验配置；`requestAiText` 接收 OpenAI messages 并返回首个有效文本响应。
- 对外统一抛出 `AiRequestError`，区分配置错误、请求失败和空文本响应；错误信息不得泄露 API Key。
- AI 客户端可注入，单元测试不访问真实模型服务。

## 节目基础信息

- `vibefm create <name> <prompt>` 创建节目空间时要求提供节目描述，供后续 AI 策划和文稿生成使用。
- `core` 在节目空间内原子写入 `info.json`，当前格式为 `{ "prompt": string }`。
- 空白 `prompt` 被视为无效输入；写入失败时删除本次新建的节目目录，避免留下不完整空间。

## 节目策划生成

- CLI 使用 `vibefm generate plan <workspace> --count <n>`，仅解析参数并输出 JSON；依赖读取、Prompt 组合、AI 返回校验和产物写入均位于 `core`。
- 生成前必须读取并校验 workspace 内的 `info.json` 和 `playlist.json`。`count` 必须是正整数且不超过歌单歌曲数量。
- Prompt 模板运行时从项目根目录 `prompts/plan.system.md` 和 `prompts/plan.user.md` 读取，便于不重新编译 CLI 即可调整模板。
- `info.json` 完整压缩为 JSON 加入 Prompt。`playlist.json` 由代码压缩为歌单 ID、名称和歌曲元组 `[id, 标题, 歌手列表, 专辑]`，不发送图片、时长、费用等无关字段，以降低 token 消耗。
- AI 只负责返回节目主题、主播风格、情绪曲线、歌曲 ID、选曲理由和单曲情绪。歌曲标题、歌手、专辑及播放序号由 `core` 根据 `playlist.json` 回填，避免模型篡改来源数据。
- 为兼容第三方 OpenAI-compatible 服务，不依赖供应商特有的 Structured Outputs；通过严格 JSON Prompt、本地解析和运行时结构校验保证结果稳定。
- AI 输出必须恰好包含指定数量的不同歌曲，所有 ID 必须来自歌单；情绪曲线必须且只能覆盖每首入选歌曲一次。首次校验失败立即返回错误，不自动重试或容错改写。
- 策划以版本化 JSON 保存到 `plan.json`。只有 AI 返回通过全部校验后才通过同目录临时文件和重命名原子替换旧产物。
- CLI 的长时间阻塞操作通过公用 `runWithBlockingNotice` 包装。进度提示写入 `stderr`，最终机器可读 JSON 继续独占 `stdout`；后续文稿、语音等 AI 命令复用同一机制。

## 节目文稿生成

- CLI 使用 `vibefm generate script <workspace>`，只负责参数解析、阻塞提示和 JSON 输出；依赖校验、Prompt 组合、AI 返回校验及产物写入位于 `core`。
- 生成前重新读取并完整校验 `info.json` 和 `plan.json`。`plan.json` 的主题、主播风格、情绪曲线、歌曲元数据、播放顺序和歌曲覆盖关系必须完整有效，不能仅因文件存在就继续生成。
- Prompt 模板运行时从项目根目录 `prompts/script.system.md` 和 `prompts/script.user.md` 读取。发送给 AI 的策划内容不包含生成时间和来源歌单等无关元数据。
- AI 直接生成 `docs/dsl.md` 定义的 RadioScript Markdown DSL，包括 `# Opening`、`# Block N`、`# Ending`、`[host]`、`[play]` 以及可选的 BGM、停顿、音效和转场事件。core 不再拆分或重组模型返回的自由文本。
- 每个 `[host]` 必须包含非空 `voice_design_prompt` 和可播读文本；每首歌前必须有主持串词。歌词内容以主题和意象解读为主，Prompt 明确禁止虚构具体歌词原句、创作背景或未经输入支持的事实。
- `[play id="..."]` 的数量、歌曲 ID 和顺序必须与 `plan.json` 完全一致。core 共用 RadioScript 解析器校验标签闭合、属性、章节位置和主持段落完整性，避免模型增删或重排歌曲。
- `script.md` 是生成事件流的规范文稿产物。后续语音和最终渲染阶段统一消费 `events.json`，不再重复解析 DSL。
- 只有 AI 返回通过全部校验后才原子替换 `script.md`；失败时保留已有文稿。

## 文稿事件流

- CLI 使用 `vibefm generate events <workspace>`，只负责参数解析和 JSON 输出；`script.md` 读取、DSL 解析、事件转换和产物写入均位于 `core`。
- `core` 使用共享 RadioScript 解析器按原始顺序解析 host、play、bgm、sfx、pause 和 transition。Markdown 标题只用于可读结构，不进入事件流。
- `events.json` 顶层直接保存 `RadioEvent[]`，作为后续 TTS、音频下载和 FFmpeg 渲染可共同消费的执行序列。
- host 事件按出现顺序获得 `host-001` 形式的稳定 ID。DSL 的百分比音量转换为 `0..1` 数值，带 `s` 的时长转换为秒数。
- 解析器拒绝未知事件、未知属性、非法单位、无效转场类型和超出范围的音量。只有整个脚本解析成功后才原子替换 `events.json`。

## 文稿分批转语音

- CLI 使用 `vibefm generate speech <workspace> [--voice <voice>] [--force]`，只负责参数解析、阻塞提示和 JSON 输出；事件读取、依赖校验、TTS 调用及产物写入位于 `core`。
- 语音阶段要求工作空间内存在完整的 `events.json`，并在调用 TTS 前校验事件数组、各事件必要字段、数值范围和 host ID 唯一性。
- 只按事件流顺序处理 `type: "host"` 的事件，每个 host 独立调用一次 TTS，并输出为 `speech/<host id>.wav`。语音阶段不再读取或解析 `script.md`。
- host 的 `voiceDesignPrompt` 作为 TTS `user` 消息中的自然语言风格指令，host 的 `text` 作为 `assistant` 消息中的待合成文本，符合 `docs/third-api/mimo-tts-api.md` 的消息约定。
- 默认跳过已经存在的 host 音频，`--force` 可重新生成。单段 TTS 失败时写入静音占位 WAV，并在 `speech/manifest.json` 中记录状态和错误，避免整批任务丢失已完成结果。

## FFmpeg 节目合成

- CLI 使用 `vibefm generate render <workspace>`，只负责参数解析、阻塞提示和 JSON 输出；依赖校验、时间线计算、滤镜图生成、FFmpeg 调用及产物写入位于 `core`。
- 渲染阶段以 `events.json` 为唯一顺序来源，并交叉校验 `speech/manifest.json` 和 `audio/manifest.json`。只接受 `synthesized` 口播与 `downloaded` 歌曲；placeholder、缺失/空文件、不安全相对路径或不可探测时长均视为依赖不完整。
- 公共素材位于项目根目录 `assets/bgm` 与 `assets/sfx`。BGM 按事件时间窗循环、裁剪和混入；radio/whoosh 转场复用 `assets/sfx` 中的同名素材。
- 所有输入先通过 `aresample`、`aformat`、`atrim` 和 `asetpts` 统一为 48 kHz、双声道、从零开始的时间戳。普通片段使用 `concat`，fade 转场使用 `acrossfade`，BGM 使用 `amix`；最终执行 `loudnorm=I=-16:LRA=11:TP=-1.5`。
- FFmpeg 通过 `spawn` 参数数组执行，不经过 shell。复杂滤镜图写入临时文件并使用 FFmpeg 8 的 `-/filter_complex` 语法载入，避免 shell 转义、路径注入和长命令行问题。当前实现与测试环境为 FFmpeg/ffprobe 8.1.1。
- 成品编码为 48 kHz 双声道、192 kbps MP3，写入 `output/program.mp3`；元数据写入 `output/manifest.json`。FFmpeg 先写同目录临时文件，成功且可被 ffprobe 读取后再重命名，失败时保留已有成品。
